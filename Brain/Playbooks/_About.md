# Playbooks — How the Team Runs Engagement Types

One playbook per engagement type: the delivery arc, checklists, and an append-only **Insights** section that accumulates lessons from real engagements.

## The pattern

- **Procedure, not facts.** Domain facts (requirements, cadences, official details) live in `Products/` with provenance tags; a playbook says *how to run the work* and cites the reference for the facts.
- **Insights are append-only, newest first, dated, and attributed** to the engagement that taught them. On every milestone (engagement completed, deliverable issued, finding closed) — or whenever the user says "playbook this" — append at least one Insight. Every engagement teaches something.
- **Failure modes earn permanent entries.** When something goes wrong, the lesson is written against the step that would have prevented it — playbooks are where near-misses become checklist lines.
- If no playbook fits a new lesson, propose a new playbook — don't force the wrong fit.
